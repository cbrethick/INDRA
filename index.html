🎯 1. PROBLEM RESTATEMENT

ISRO is not asking for a rainfall forecasting model. They are asking for a STATEFUL, CONTINUOUSLY UPDATING VIRTUAL REPLICA of a piece of India's climate that can:

  1. ingest national datasets (IMD + INSAT/MOSDAC),
  2. hold a "current state" of the climate over a pilot region,
  3. predict forward in time,
  4. be perturbed by a user ("what if SST rises 1.5 degC") and show the downstream effect,
  5. be inspected visually on a map/dashboard,
  6. be validated honestly against real IMD observations.
A model alone satisfies (c). Everything else is what separates a "ML project" from a "Digital Twin," and the rubric is explicitly built to reward the difference (see Section 17).

Digital Twin concept: continuously updating virtual replica, not just a forecast model.

♟️ 2. STRATEGIC POSITIONING — WHAT ACTUALLY WINS THIS PROBLEM

Three things separate a top submission from the pack:

  1. (1) A STATE OBJECT, not just a model. If you cannot point to a concrete data structure called "the current twin state" that updates over time, you don't have a digital twin — you have a forecasting script with a nice UI. Section 6 defines this explicitly.
  2. (2) HONEST VALIDATION AGAINST A BASELINE, WITH UNCERTAINTY. The number that wins the slide is skill over a naive baseline (climatology/persistence), reported alongside a calibrated uncertainty interval — not a bare RMSE, and not a vague "accuracy."
  3. (3) A SCENARIO MODULE THAT IS CLEARLY LABELLED AS AN AI EMULATION, NOT A PHYSICAL CLIMATE MODEL. Correctly framing it ("this is a learned statistical emulator for rapid scenario exploration, not a substitute for a GCM") is itself a point under "Problem Understanding & Clarity."

Build order, in priority: working baseline → working twin-state object → working API → working map UI → scenario engine → validation page → explainability/risk-index/advisory polish → LLM copilot (only if everything above is solid). A finished core loop beats five half-built fancy modules, every time.

📊 4. DATA STRATEGY

4.1 Sources and years (locked decision)

Mandatory (download manually from IMD dropdowns, Day 0, first task):

  • Gridded Rainfall, 0.25×0.25 deg, 2015–2024 (10 files) — Grid: 135×129, float32, lon 66.5E–100.0E, lat 6.5N–38.5N
  • Maximum Temperature, 1.0×1.0 deg, 2015–2024 (10 files) — Grid: 31×31, float32, lon 67.5E–97.5E, lat 7.5N–37.5N
  • Minimum Temperature, 1.0×1.0 deg, 2015–2024 (10 files) — Same grid as max temp.

Total: 30 binary files. Known, bounded, achievable on Day 0.

Day-0 PARALLEL task (start the clock immediately, don't wait): Register for MOSDAC access and request INSAT LST (product 3RIMG_L2B_LST). If approval lands within the hackathon window, LST becomes a 4th input channel — a real differentiator. If approval has not landed by Hour 8, proceed on the IMD-only path without it. This is a deliberate fallback, not a failure — document it as such on the future-work slide.

Free static layers (cheap, reuse your existing Earth Engine workflow):

  • DEM (elevation) — helps the model learn orographic effects, one GEE export.
  • NDVI — cheap proxy for land-surface/vegetation state, also a single GEE export. Skip land cover and soil moisture for the MVP; mention them as future-work layers instead.

4.2 Why 10 years, not 124

124 years of daily gridded data is hundreds of GB once decoded and is irrelevant to a PoC over a pilot region. Ten years (3,650+ daily grids) is enough to learn a seasonal cycle and short-lag persistence structure, and is small enough to preprocess on a laptop. State this scoping decision explicitly — judges reward a justified decision more than raw data volume.

4.4 IMD binary decoder (native Python)

Module: backend/preprocessing/imd_binary_decoder.py

Responsibilities:

  1. Detect leap year (365 vs 366 records) from the requested year.
  2. Read the flat binary file at the documented record size (rainfall: 135×129×4 bytes/day; temp: 31×31×4 bytes/day).
  3. Decode float32 values into a NumPy array, shape [days, lat, lon].
  4. Generate lat/lon coordinate arrays from the documented grid origins.
  5. Handle missing-value flags: -999 for rainfall, 99.9 for temperature — never silently zero-fill.
  6. Wrap into an xarray.DataArray with a proper daily time index.
  7. Export to NetCDF/Zarr once per year — raw binary decoded only once.

4.5 Preprocessing pipeline

Raw binary → Python decoder (4.4) → xarray with coordinates → clip to pilot region → regrid temperature (31×31, 1deg) onto the rainfall grid (135×129, 0.25deg) via bilinear interpolation → missing value handling → normalize per-channel (store the scaler) → stack into daily tensors → add static DEM/NDVI layers → validate (Section 4.6) → write NetCDF/Zarr to processed/.

⚠️ Regridding the 31×31 temperature grid onto the 135×129 rainfall grid is the single most common bug source in this entire pipeline. Budget real debugging time for this exact step, and sanity-check by plotting before/after regrid maps side by side before moving on.

4.6 Data integrity validation (before training starts)

Check per year/file: file size matches expected record count, day count matches leap-year rule, grid dimensions match spec, missing-value rate is within a sane range, latitude/longitude ordering is correct (rainfall data is stored south-to-north). Write a validation_report.json — this doubles as evidence of rigor in your data-handling slide.

4.7 Daily tensor schema

Shape: [H, W, C]

Core channels (always): rainfall_mm, max_temp_C, min_temp_C, elevation_m, ndvi

Stretch channel (if MOSDAC lands in time): lst_C

🧠 5. MODELING STRATEGY — PHASED, ESCALATING TOWARD DEEP LEARNING

Build in this order. Each tier is independently demoable — stop at whichever tier the clock allows, but Tier 2 should be a real attempt, not an afterthought, since judges expect a deep-learning component.

TIER 0 — Baselines (build first, ~1–2 hours, never skip)

  • Persistence: tomorrow = today.
  • Climatology: tomorrow = historical mean for that day-of-year.

These exist purely to give every later tier something honest to beat.

TIER 1 — Gradient-boosted core model (the guaranteed working result)

XGBoost or LightGBM, per-pixel, with lag features (t-1, t-2, t-3, day-of-year, lat, lon, elevation, ndvi) → predicts rainfall_mm and temp for t+1. Trains in minutes, no GPU dependency, gives clean SHAP feature attributions for free (Section 10), and is the lowest-risk path to a working, validated prediction by hour 10–12. This is your fallback-safe primary result if nothing else finishes.

TIER 2 — ConvLSTM (the deep-learning component — attempt this seriously)

ConvLSTM over the [H,W,C] grid sequence → predicts the next 1–3 day grids directly, capturing spatial structure (rain systems moving across the grid) that Tier 1 cannot. Start this as soon as Tier 1 is validated and the API is wired to it — do not let Tier 2 block the rest of the pipeline from being demoable. If Tier 2 underperforms or doesn't converge in time, present it honestly as "in progress, partial results" rather than hiding it.

TIER 3 — Stretch refinements

  • Quantile regression or ensemble spread → prediction uncertainty intervals (feeds Section 10's confidence layer).
  • A lightweight temporal-attention/Informer-style refinement layer on top of Tier 2, if a team member has the time and GPU access.
Best presentation if time allows: show Tier 1 (XGBoost) as the production-ready, fully validated result, and Tier 2 (ConvLSTM) as the spatial-context research direction with its own validation numbers. "Simple model that works + advanced model we pushed on" is a stronger technical narrative than either alone.

Derived output (cheap, add once Tier 1 predictions exist): a CLIMATE RISK INDEX, a simple weighted combination of rainfall anomaly and temperature anomaly against the 2015–2024 climatology, surfaced as a single 0–100 score per grid cell/district.

🔗 6. DIGITAL TWIN CORE — WHAT MAKES THIS A TWIN, NOT JUST A MODEL

TWIN STATE OBJECT (a real, inspectable thing — show it in the demo):

{
            region_id: "KL_pilot",
            timestamp: "2024-09-12",
            observed: { rainfall, max_temp, min_temp, ndvi, [lst] },
            predicted: { rainfall_t1, temp_t1, ... },
            uncertainty: { rainfall_ci, temp_ci },        <- Tier 3
            risk_index: 0-100,                             <- Section 5
            version: int,
            provenance: "IMD-gridded-v1 + XGBoost-v2 + ConvLSTM-v1"
        }

UPDATE LOOP (runs whenever new data lands, or on a schedule for the demo): new daily file arrives → preprocess → blend into canonical state (replace observed values, recompute predicted/uncertainty/risk_index) → increment version → persist snapshot → push to API/dashboard.

THREE DISTINCT MODES your dashboard must visibly expose as three different things:

  1. REPLAY (time-travel through stored history, 2015–2024)
  2. FORECAST (canonical state's prediction for t+1, t+2, t+3)
  3. SCENARIO (a forked, non-canonical state created by the scenario engine, Section 8 — never overwrites the canonical state)

Store snapshots as lightweight JSON/Parquet keyed by region+date+version. This versioning IS your honest, lightweight substitute for a full MLOps stack — you can say "twin state v47, here is v48" out loud in the demo without needing a real model registry behind it.

✅ 7. VALIDATION FRAMEWORK

Split (walk-forward, no leakage): Train: 2015–2022 | Validation: 2023 | Test: 2024 (touch once)

Continuous metrics: RMSE, MAE, R-squared, Pearson correlation — report all four, per variable.

Event metrics (rain-day vs no-rain-day, threshold ≥2.5mm): POD, FAR, CSI.

Uncertainty: if Tier 3 is built, report interval coverage — a calibrated-but-wide interval is more credible than an overconfident point estimate.

THE NUMBER THAT WINS THE SLIDE: skill score against Tier 0 baseline —
Skill = 1 − (RMSE_model / RMSE_climatology)
Report per-variable, per-tier (Tier 1 skill, Tier 2 skill). A skill score of 0.25–0.35 over climatology for next-day rainfall in a small pilot region is believable and defensible — do not inflate it.

🎛️ 8. SCENARIO ENGINE (formerly "what-if simulator")

Two entry points into the same underlying mechanism:

(a) MANUAL SLIDERS — user adjusts temperature delta, rainfall delta, (stretch: SST/LST delta).

(b) PRESET SCENARIOS — one click loads a pre-defined parameter combination:

  • Heatwave (+3–4°C max temp, −humidity proxy)
  • Heavy Rain Event (+40–60% rainfall over 2–3 days)
  • Weak Monsoon (−20–30% JJAS rainfall)
  • Strong Monsoon (+15–25% JJAS rainfall)
  • El Niño-like (warmer SST/LST signal, drier bias) [if LST channel available]
  • La Niña-like (cooler SST/LST signal, wetter bias)

Presets are just named parameter sets calling the same mechanism as (a) — low extra engineering cost, high demo impact.

Mechanics (identical for both entry points):

  1. Take the CURRENT canonical twin state's input tensor.
  2. Apply the perturbation in real, un-normalized units, then re-apply the stored scaler (Section 4.5).
  3. Run inference with the frozen, already-trained Tier 1 (and Tier 2 if ready) model on the perturbed tensor.
  4. Store the result as a new SCENARIO state (Section 6) — never overwrites canonical state.
  5. Dashboard shows a side-by-side delta map (baseline forecast vs. scenario forecast) plus a one-line plain-English explanation pulled from SHAP attributions.

Pre-bake one working preset scenario for the live demo — don't improvise live.

MANDATORY UI DISCLAIMER (say it out loud in the pitch too): this is a learned statistical emulator trained on historical IMD/INSAT patterns for rapid scenario exploration — not a substitute for a physical General Circulation Model. Stating this clearly is a strength under "Problem Understanding & Clarity," not a weakness.

🖥️ 9. DASHBOARD / VISUALIZATION

Build in this order:

  1. TODAY screen (home) — current observed state, tomorrow's forecast, risk index, confidence, last-update timestamp. Modeled deliberately on a familiar "weather app" mental model so judges grasp it in 5 seconds.
  2. Map view — pilot region, layer toggle: Observed / Predicted / Scenario, for rainfall, temperature, and risk index.
  3. Timeline scrubber — drag through 2015–2024 (Replay mode).
  4. Validation page — metrics table (Section 7) + predicted-vs-actual scatter plot.
  5. Scenario panel — sliders + presets (Section 8) + delta map.
  6. District drill-down — click a cell/district → trend chart + risk flag + AI-generated advisory line (Section 10).
  7. (Stretch) Climate Copilot panel (Section 10).

Tech: React + Leaflet (fastest path to a working choropleth grid in a hackathon) + Recharts for charts. FastAPI backend serving twin-state JSON and predict/scenario endpoints.

💡 10. INNOVATION FEATURES — PRIORITIZED, ONLY AFTER THE CORE LOOP WORKS

HIGH IMPACT, LOW EFFORT (do these first, in this order):

  • SHAP feature attribution per Tier 1 prediction (near-free with XGBoost) — shown as a short "why this prediction" line.
  • Skill score vs baseline, displayed prominently (already computed, Section 7).
  • Climate Risk Index (Section 5) surfaced on the Today screen and map.
  • District click → trend + risk flag + one-line advisory, e.g.: "Flood Risk: High | Reason: rainfall predicted >120mm, temp 29°C | Confidence: 91%"
  • Scenario presets (Section 8(b)).

MEDIUM IMPACT, MEDIUM EFFORT (only if the above finish early):

  • Prediction uncertainty intervals (Tier 3, Section 5) shown as a shaded band, not just a point estimate.
  • Attention/saliency visualization on ConvLSTM (Tier 2), if it trains successfully — shows spatially WHERE the model is "looking."

STRETCH (attempt only after every section above is fully working and demoable):

  • CLIMATE COPILOT: a minimal templated assistant, not a full RAG system. Implementation: take the current twin-state JSON (Section 6) + a short canned context paragraph about the pilot region, feed both into a single LLM prompt with the user's question, return the answer. This answers "why is rainfall decreasing in [region]" or "compare 2023 and 2024" using only the structured numbers already in the twin state — a thin natural-language wrapper around data you already have.

DELIBERATELY NOT ATTEMPTED for a 48-hour PoC:

  • A real model registry/experiment-tracking stack
  • Land cover and soil moisture layers
  • A 3D globe view
  • Multi-region national-scale serving

All four are legitimate roadmap items, none are achievable credibly in 48 hours alongside everything above.
